公開前の記事に秘匿情報(ID・パスワード・トークン・ARN等)が含まれていないかを3層でチェックするワークフローです。プレースホルダー規約の正データは `docs/zenn-rules.md` の「秘匿すべき情報の扱い」(このファイルには複製しない)。

パスはすべてリポジトリルート基準です。

## 0. 対象特定

- 引数があればそのファイル、`--all` なら全記事監査
- 引数が無ければ `articles/` で最も最近更新されたmdファイルを候補としてユーザーに確認する

## 1. 機械スキャン(ドライバ)

```bash
node .claude/skills/security-check/scripts/security-scan.mjs <file>
node .claude/skills/security-check/scripts/security-scan.mjs --all    # 全記事監査
```

- 検出レベル: **high**(APIキー・トークン・秘密鍵・JWT・認証付きURL等、ほぼ確実な秘匿情報) / **medium**(ARN・UUID・32桁hex・資格情報代入・URLクエリトークン等、固有IDの可能性) / **info**(メール・パブリックIP等、文脈確認が必要)
- **git履歴スキャン**: 現行版でマスク済みでも、マスク前の実値が公開履歴に残っていれば `[HISTORY]` として検出する(コミット済みファイルのみ対象。`--no-history` でスキップ)
- **画像列挙**: 記事が参照する画像を列挙する。スクリーンショット内の秘匿情報はregexでは検出できないため、手順2で必ず確認する
- `{名前}` / `<説明>` / `xxxx` / `****` / `${{ secrets.X }}` / `process.env.X` 等のプレースホルダーは検出対象外
- 終了コード: high/mediumが1件以上(履歴含む)なら1、なければ0。`--json` で構造化出力

## 2. 文脈調査(サブエージェント)

`security-reviewer` サブエージェントを起動し、記事パスと手順1で列挙されたローカル画像のパス一覧を渡す。証拠収集タスクのため、エージェント定義どおり Sonnet 5(`model: sonnet`)で実行する(モデルを上書きしない)。

- regexで型が決められない「実物らしい固有値」・個人/組織情報・スクリーンショットの写り込みを調べ、伏せ字付きの証拠として返す
- 外部URL画像(zenn-user-upload等)はサブエージェントも取得できないため「未確認」として返る → ユーザーに目視確認を依頼する

## 3. 統合判定(あなた自身が行う)

機械スキャンとサブエージェントの証拠の統合・最終判定は抽象度の高い意思決定のため、サブエージェントに委譲せずあなた自身(Fable 5)が行う。3分類で提示する:

- **[公開ブロッカー]**: high検出 / 実物と確認できた固有値 / 画像内の秘匿情報の写り込み
- **[要確認]**: medium・infoのうち実物か判断できないもの / 未確認の外部画像 / `[HISTORY]` 検出
- **[問題なし]**: プレースホルダー化済み / 意図的な公開と確認できたもの(公開DNSで誰でも引ける値など、そもそも公開情報であるものを含む)

修正案は `docs/zenn-rules.md` のプレースホルダー規約(`{名前}` / `< 説明 >` 形式、例: `arn:< userのid >/xxxxx`)に従って提示し、適用はユーザーに確認してから行う。

## 4. 履歴に実値が残っていた場合(`[HISTORY]` 検出時)

公開リポジトリのgit履歴は誰でも参照できるため、記事の修正だけでは対処にならない。優先順位を添えてユーザーに提示する:

1. **値のローテーション(無効化・再発行)が第一候補**。履歴の書き換えは、既にクローン/フォークされた複製には効かない
2. ローテーション不能な値(個人情報等)に限り、履歴書き換え(git filter-repo等)を補助手段として案内する。force pushを伴うため必ずユーザーの明示的な判断を仰ぐ
3. リスクの低い識別子(削除済みリソースのID、単体では悪用できないID等)は「対応不要」の判断もあり得る。判断材料を添えてユーザーに委ねる

## Gotchas(実際に実行して分かったこと)

- このリポジトリでは commit 741ba0f 以前の公開履歴に D1 `database_id` の実値(UUID)が残っている(現行版はマスク済み)。履歴スキャンはこれを実際に検出する
- hookの `.claude/scripts/scan-secrets.mjs` は編集時の即時ブロック用(軽量)で、本ドライバとは役割が別。本ドライバはhookを置き換えない
- 記事に頻出する `next@16.2.3`(パッケージ@バージョン)、`Chrome/147.0.0.0`(UAバージョン)、`process.env.X`(環境変数参照)は誤検知しないよう抑制済み。新たな誤検知を見つけたらドライバの `PLACEHOLDER_HINTS` か各ルールの `suppress` に追加する
- ドライバのルールを変更したら `node .claude/skills/security-check/scripts/security-scan.mjs --self-test` で回帰確認する(真陽性11ルール+誤検知パターン集を内蔵。PASSで終了コード0)
- 履歴スキャンはregexルール(high/medium)のみ。ネームサーバー名のような「型のない値」の履歴残存は検出できない(実例: Cloudflare NS名は履歴に残っているが、DNSで公開される情報のためリスクは低い)
- 全記事+全履歴のスキャンでも数秒で終わる。公開前は `--all` を惜しまない
